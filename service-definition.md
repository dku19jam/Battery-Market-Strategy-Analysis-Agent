# Battery Market Strategy Analysis Agent Service Definition

## 1. 문서 목적

본 문서는 `배터리 시장 전략 분석 Agent` 프로젝트의 통합 설계 문서이다.  
문서의 목적은 다음 두 가지를 동시에 만족하는 것이다.

- 과제 제출용 설계 문서로서 프로젝트 목표, 구조, 처리 흐름, 산출물을 명확히 설명한다.
- 개발자 구현용 명세 문서로서 입력, 출력, 에이전트 책임, 실패 처리, 산출물 규격을 구체적으로 정의한다.

본 프로젝트는 범용 산업 분석 플랫폼이 아니라, 과제 수행을 위한 `전용 비교 분석 서비스`로 정의한다.

## 2. 프로젝트 목표

서비스의 목표는 `LG에너지솔루션`과 `CATL`의 포트폴리오 다각화 전략을 비교 분석하고, 이를 바탕으로 최종 전략 분석 보고서를 자동 생성하는 것이다.

구체적으로 시스템은 다음을 수행해야 한다.

- 배터리 시장 환경 변화와 관련된 근거 자료를 수집한다.
- 두 기업의 포트폴리오 다각화 전략과 핵심 경쟁력을 정리한다.
- 두 기업의 전략 차이점과 강약점을 데이터 기반으로 비교한다.
- SWOT 분석과 종합 시사점을 도출한다.
- 최종 보고서를 한국어 `Markdown`과 `PDF` 형식으로 자동 생성한다.

## 3. 프로젝트 범위

### 3.1 포함 범위

- `LG에너지솔루션`과 `CATL`의 전략 비교 분석
- 로컬 코퍼스 기반 RAG 검색
- 제한적 웹 검색 기반 보완 수집
- 근거 정리 및 중복 제거
- 기업별 전략 분석
- 기업 간 비교 분석
- SWOT 분석
- 종합 시사점 도출
- 참고문헌 자동 정리
- 최종 보고서 `Markdown + PDF` 생성
- 중간 산출물 저장

### 3.2 제외 범위

- 범용 기업 비교 플랫폼
- 웹 UI 또는 대시보드
- 사용자가 다양한 기업 조합을 직접 선택하는 범용 입력 시스템
- 실시간 협업 기능
- 사람의 수동 편집을 전제로 한 반자동 보고서 작성 도구

## 4. 사용자와 사용 방식

이 서비스는 `CLI/스크립트 실행형`으로 동작한다.

- 사용자는 터미널에서 서비스를 실행한다.
- 사용자는 최소 입력만 제공한다.
- 기본 비교 대상은 `LG에너지솔루션`과 `CATL`로 고정한다.
- 기본 보고서 언어는 `한국어`로 고정한다.
- 과제 전용 서비스이므로 입력 다양성보다 `재현성`, `안정성`, `자동화 수준`을 우선한다.

## 5. 입력과 출력

### 5.1 입력

사용자 입력은 최소화한다.

- 사용자는 실행 시 비교 주제만 지정하거나, 기본 과제 주제를 그대로 사용할 수 있다.
- 기업명은 과제 범위에 맞게 고정한다.
- 실행 옵션은 최소한으로 유지하며, 상세 설정은 내부 정책으로 처리한다.

예시 입력 개념은 다음과 같다.

- 배터리 시장 전략 비교
- 전기차 캐즘 대응 전략 비교
- 포트폴리오 다각화 전략 비교

### 5.2 출력

최종 산출물은 다음을 포함한다.

- 최종 전략 분석 보고서 `Markdown`
- 최종 전략 분석 보고서 `PDF`
- 검색 결과 로그
- 수집 근거 정리 파일
- 기업별 분석 중간 결과
- 비교 분석 결과
- 참고문헌 후보 목록 및 최종 참고문헌 목록
- 실행 로그 및 재시도 로그

## 6. 성공 기준

서비스는 아래 기준을 만족해야 한다.

- 최종 보고서는 `LG에너지솔루션`과 `CATL` 비교를 중심으로 구성되어야 한다.
- 보고서는 데이터 근거 기반으로 작성되어야 한다.
- 보고서는 한국어로 일관되게 작성되어야 한다.
- 최종 산출물은 `Markdown`과 `PDF` 두 형식 모두 생성되어야 한다.
- 참고문헌은 실제 사용한 자료만 포함해야 한다.
- 동일한 로컬 코퍼스 기준에서 결과 재현성이 확보되어야 한다.
- 일부 단계 실패가 발생해도 제한된 재시도 후 가능한 범위의 보고서를 생성해야 한다.

## 7. 전체 처리 흐름

서비스의 기본 흐름은 다음과 같다.

1. 사용자가 CLI를 통해 서비스를 실행한다.
2. 실행 런타임이 초기 상태를 준비하고 시작 agent를 호출한다.
3. LG Retrieval Agent가 로컬 코퍼스 기반 RAG 검색과 제한적 웹 검색을 수행하고, 충분한 근거가 확보될 때까지 자기 범위 내에서 질의 재작성과 재검색을 반복할 수 있다.
4. LG Evidence Curation Agent가 LG 관련 근거를 정리하고, 필요 시 추가 근거가 필요하다는 판단을 LG Retrieval Agent에 다시 전달한다.
5. LG Analysis Agent가 LG에너지솔루션 전략 분석 결과를 생성하고, 근거 부족 시 LG 라인 내부에 보완 요청을 반환할 수 있다.
6. CATL Retrieval Agent가 로컬 코퍼스 기반 RAG 검색과 제한적 웹 검색을 수행하고, 충분한 근거가 확보될 때까지 자기 범위 내에서 질의 재작성과 재검색을 반복할 수 있다.
7. CATL Evidence Curation Agent가 CATL 관련 근거를 정리하고, 필요 시 추가 근거가 필요하다는 판단을 CATL Retrieval Agent에 다시 전달한다.
8. CATL Analysis Agent가 CATL 전략 분석 결과를 생성하고, 근거 부족 시 CATL 라인 내부에 보완 요청을 반환할 수 있다.
9. 두 기업의 분석 결과가 준비되면 Comparison Evaluation Agent가 비교 평가를 수행하고, 필요 시 기업별 분석 보완을 요청할 수 있다.
10. 실제 사용 근거가 확정되면 Reference Agent가 최종 참고문헌을 생성한다.
11. 비교 결과와 참고문헌이 준비되면 Report Generation Agent가 구조화된 최종 Markdown 보고서를 생성한다.
12. Markdown 결과를 PDF로 변환한다.
13. 중간 산출물과 로그를 함께 저장한다.

## 8. 멀티 에이전트 아키텍처

### 8.1 기본 구조

멀티 에이전트 구조는 `Distributed pattern`을 기본으로 사용한다.

- 중앙 의사결정용 Supervisor Agent는 두지 않는다.
- 각 에이전트는 단일 책임 원칙에 따라 분리한다.
- 각 에이전트는 공유 workflow state를 읽고 자신의 산출물을 기록한다.
- 각 에이전트는 자신의 역할 범위 안에서 `도구 사용`, `재시도`, `부분 종료`, `다음 단계 handoff`를 부분적으로 판단한다.
- 다음 단계 실행 여부는 중앙 판단 노드가 아니라 agent의 출력 상태와 handoff 결과로 결정된다.
- 실행 런타임은 그래프 실행과 상태 전달만 담당하며, 분석 판단 자체에는 관여하지 않는다.

### 8.2 에이전트 구성

#### LG Retrieval Agent

- LG에너지솔루션 관련 자료만 로컬 코퍼스에서 검색한다.
- LG에너지솔루션 관련 웹 보완 검색을 수행한다.
- 검색 결과와 근거 문서 식별자를 LG 전용 결과로 반환한다.

#### LG Evidence Curation Agent

- LG에너지솔루션 관련 자료를 주제별로 분류한다.
- LG 전용 근거에서 중복 자료를 제거한다.
- 신뢰도 낮은 자료를 제외하거나 우선순위를 낮춘다.
- LG 분석에 사용 가능한 근거 묶음을 생성한다.

#### LG Analysis Agent

- LG에너지솔루션의 포트폴리오 다각화 전략을 정리한다.
- LG에너지솔루션의 핵심 경쟁력과 리스크를 정리한다.
- LG 전용 기업 분석 결과를 생성한다.

#### CATL Retrieval Agent

- CATL 관련 자료만 로컬 코퍼스에서 검색한다.
- CATL 관련 웹 보완 검색을 수행한다.
- 검색 결과와 근거 문서 식별자를 CATL 전용 결과로 반환한다.

#### CATL Evidence Curation Agent

- CATL 관련 자료를 주제별로 분류한다.
- CATL 전용 근거에서 중복 자료를 제거한다.
- 신뢰도 낮은 자료를 제외하거나 우선순위를 낮춘다.
- CATL 분석에 사용 가능한 근거 묶음을 생성한다.

#### CATL Analysis Agent

- CATL의 포트폴리오 다각화 전략을 정리한다.
- CATL의 핵심 경쟁력과 리스크를 정리한다.
- CATL 전용 기업 분석 결과를 생성한다.

#### Comparison Evaluation Agent

- LG와 CATL의 독립 분석 결과를 입력으로 받는다.
- 두 기업의 전략적 차이점을 비교한다.
- 강점과 약점을 도출한다.
- SWOT 분석을 수행한다.
- 종합 시사점을 정리한다.

#### Report Generation Agent

- 보고서 목차에 맞춰 최종 내용을 조립한다.
- 문체, 구조, 서술 일관성을 정리한다.
- Markdown 보고서를 생성한다.

#### Reference Agent

- 실제 사용한 자료만 추적한다.
- 자료 유형별 참고문헌 포맷을 적용한다.
- 최종 `REFERENCE` 섹션을 생성한다.

### 8.3 실행 런타임

실행 런타임은 에이전트가 아니며, 아래 역할만 수행한다.

- workflow 시작 시 공유 state 초기화
- 그래프에 정의된 입력/출력 전달
- agent handoff 결과에 따라 다음 agent 실행
- 로그 및 중간 산출물 저장

## 9. 워크플로 정의

문서 요구사항에 맞춰 Agentic Workflow는 아래 항목을 포함해야 한다.

### 9.1 Goal

- LG에너지솔루션과 CATL의 포트폴리오 다각화 전략을 비교 분석한 최종 보고서를 자동 생성한다.

### 9.2 Criteria

- 두 기업의 전략과 핵심 경쟁력이 모두 정리되어야 한다.
- 전략 비교와 SWOT 분석이 포함되어야 한다.
- 시장 배경과 종합 시사점이 포함되어야 한다.
- 보고서 형식과 참고문헌 형식이 요구사항을 충족해야 한다.
- 최종 결과물은 Markdown과 PDF 두 형식으로 생성되어야 한다.

### 9.3 Task

- LG 자료 수집
- LG 근거 정리
- LG 전략 분석
- CATL 자료 수집
- CATL 근거 정리
- CATL 전략 분석
- 기업 간 비교 평가
- SWOT 분석
- 종합 시사점 생성
- 참고문헌 생성
- Markdown/PDF 보고서 생성

### 9.4 Control Strategy

- Distributed pattern 기반 agent-local 제어
- agent별 산출물 기록 후 handoff와 상태값에 따라 다음 단계 진행
- 근거 부족 시 해당 agent가 자기 역할 범위 내에서 제한된 재시도 또는 보완 요청 수행
- 재시도 후에도 실패 시 부분 보고서 생성

### 9.5 Structure

- Distributed pattern 기반 멀티 에이전트 구조
- LG 전용 검색/정리/분석 라인과 CATL 전용 검색/정리/분석 라인을 분리
- 각 라인은 agent 간 handoff와 보완 요청을 통해 독립적으로 수렴한다
- 두 기업의 분석 결과가 모두 준비되면 Comparison Evaluation Agent가 두 결과를 통합 평가한다
- Reference Agent와 Report Generation Agent는 통합 결과와 사용 근거를 기반으로 최종 산출물을 생성한다

## 10. 데이터 수집 및 RAG 설계

### 10.1 로컬 코퍼스 정의

로컬 코퍼스는 다음을 포함한다.

- 사전에 수집한 PDF 문서
- 사전에 저장한 웹 문서
- 실행 전에 정리된 보고서, 기사, 분석 자료

로컬 코퍼스는 본 서비스의 기본 분석 근거 저장소로 사용한다.

### 10.2 RAG 정책

- Retrieve 기능은 `Agentic RAG` 기반으로 설계한다.
- 분석 대상 문서는 총 `100페이지 이내`로 제한한다.
- RAG는 주된 근거 수집 방식으로 사용한다.
- 검색 결과는 기업명, 전략 키워드, 시장 변화 키워드 기준으로 분류한다.
- 검색 근거는 반드시 중간 산출물로 저장한다.

### 10.3 재현성 정책

- 분석은 `재현성 우선`으로 설계한다.
- 로컬 코퍼스를 핵심 근거로 사용한다.
- 웹 검색은 주된 분석 소스가 아니라 보완 용도로만 사용한다.
- 동일한 코퍼스 기준에서 결과를 다시 생성할 수 있어야 한다.

## 11. 웹 검색 설계와 편향 완화 전략

웹 검색은 허용하되, 보완 검색으로 제한한다.

### 11.1 웹 검색 목적

- 로컬 코퍼스로 부족한 최신 정보 보완
- 특정 전략 이슈에 대한 추가 근거 확보
- LG와 CATL 각각의 최근 사업 방향 확인

### 11.2 편향 완화 원칙

- 긍정적 기사만 반복 인용하지 않는다.
- 한 기업에 유리한 자료만 집중적으로 수집하지 않는다.
- 전략 성과뿐 아니라 리스크, 약점, 제약 요인도 함께 수집한다.
- 가능하면 서로 다른 성격의 출처를 혼합한다.
- 최종 분석 시 과장 표현보다 근거 중심 진술을 우선한다.

### 11.3 웹 검색 사용 제한

- 웹 검색 결과는 무제한으로 확장하지 않는다.
- 로컬 코퍼스로 충분한 경우 웹 검색을 생략할 수 있다.
- 웹 검색 결과는 저장 후 검토 가능한 형태로 남긴다.

## 12. 모델 및 기술 스택 요구사항

### 12.1 LLM 제공자

- LLM 제공자는 `OpenAI`로 고정한다.

### 12.2 기본 모델

- 기본 생성 및 분석 모델은 `gpt-4o-mini`를 우선 사용한다.
- 비용 효율성과 응답 품질의 균형을 고려한 선택으로 정의한다.

### 12.3 임베딩 모델

- 임베딩 모델은 반드시 `오픈소스`를 사용해야 한다.
- 임베딩 모델은 로컬 코퍼스 RAG 검색에 사용한다.
- 경제성을 고려해 추론 비용보다 저장 및 검색 효율을 우선한다.

## 13. 보고서 요구사항

### 13.1 필수 포함 내용

보고서에는 아래 항목이 포함되어야 한다.

- 시장 배경
- 각 기업별 포트폴리오 다각화 전략
- 각 기업별 핵심 경쟁력
- 핵심 전략 비교
- SWOT 분석
- 종합 시사점

### 13.2 목차 규칙

목차는 자유롭게 정의할 수 있으나 다음을 반드시 포함해야 한다.

- 맨 앞 `SUMMARY`
- 맨 뒤 `REFERENCE`

### 13.3 SUMMARY 규칙

- `SUMMARY`는 전체 전략 분석 보고서의 핵심 요약이어야 한다.
- 개요용 장식 페이지가 아니라 실제 핵심 내용을 요약해야 한다.
- 길이는 `0.5페이지 이내`로 제한한다.

### 13.4 REFERENCE 규칙

- 실제로 활용한 자료만 포함해야 한다.
- 사용하지 않은 후보 자료는 최종 참고문헌에서 제외한다.

## 14. 참고문헌 포맷 요구사항

최종 참고문헌은 자료 유형별 형식을 따른다.

### 14.1 기관 보고서

- `발행기관 (YYYY). 보고서명. URL`

### 14.2 학술 논문

- `저자 (YYYY). 논문제목. 학술지명, 권(호), 페이지.`

### 14.3 웹페이지

- `기관명 또는 작성자 (YYYY-MM-DD). 제목. 사이트명, URL`

## 15. 중간 산출물 요구사항

최종 결과뿐 아니라 중간 결과도 모두 저장해야 한다.

중간 산출물 예시는 다음과 같다.

- 로컬 코퍼스 검색 결과
- 웹 검색 결과
- 문서별 발췌 근거
- 기업별 분석 메모
- 비교 분석 메모
- SWOT 초안
- 참고문헌 후보 목록
- 재시도 기록
- 오류 및 실패 로그

이 정책의 목적은 다음과 같다.

- 분석 근거 추적 가능성 확보
- 결과 검토 가능성 확보
- 재현성 강화
- 디버깅 용이성 확보

## 16. 실패 대응 및 재시도 정책

### 16.1 기본 원칙

- 필수 섹션 근거가 부족하면 즉시 최종 생성을 강행하지 않는다.
- 제한된 횟수 안에서 재검색 또는 재분석을 수행한다.
- 재시도 후에도 근거가 충분하지 않다면 부분 보고서를 생성한다.

### 16.2 재시도 순서

재시도는 아래 순서로 수행한다.

1. 기업별 로컬 코퍼스 재검색
2. 기업별 질의 재작성 후 재검색
3. 기업별 제한적 웹 검색 기반 보완 수집
4. 기업별 보완 근거를 반영한 재분석
5. 두 기업 분석 결과가 확보되면 비교 평가 재실행

### 16.3 부분 보고서 생성 정책

- 일부 섹션의 근거가 부족하더라도 전체 결과를 가능한 범위에서 생성한다.
- 부족한 섹션은 비워두지 않고 `근거 부족으로 제한된 분석`임을 명시한다.
- 분석 불확실성이 큰 섹션은 단정적인 결론을 피한다.

### 16.4 전체 실패 조건

아래와 같은 경우는 전체 실패로 처리할 수 있다.

- `SUMMARY` 생성이 불가능한 경우
- `REFERENCE` 생성이 불가능한 경우
- 두 기업 비교의 중심 근거가 전반적으로 붕괴한 경우

## 17. 비기능 요구사항

### 17.1 재현성

- 동일 코퍼스 기준에서 유사한 결과를 다시 생성할 수 있어야 한다.

### 17.2 추적 가능성

- 각 주장에 사용된 근거와 출처를 추적할 수 있어야 한다.

### 17.3 일관성

- 보고서 전반의 용어, 문체, 섹션 구조가 일관되어야 한다.

### 17.4 안정성

- 일부 단계 실패가 발생해도 전체 파이프라인이 즉시 중단되지 않아야 한다.

### 17.5 비용 효율성

- LLM은 `gpt-4o-mini` 중심으로 운용한다.
- 임베딩은 오픈소스 모델을 사용한다.

과제 제출 기준으로 아래 내용이 문서 또는 구현 산출물에 포함되어야 한다.

## 20. 향후 확장 가능성

초기 구현은 과제 전용 구조로 제한하지만, 아래 확장 가능성은 열어둔다.

- 현재 distributed pattern을 이벤트 기반 분산 실행 구조로 확장
- 비교 대상 기업 수 확장
- 산업 범위 확장
- 웹 UI 추가
- 모델 교체 또는 다중 모델 전략 추가

단, 위 항목은 현재 구현 범위에는 포함하지 않는다.

## 21. 현재 기준에서 확정된 설계 결정

- 과제 제출용 설계 문서와 개발자 구현 명세를 하나의 문서로 통합한다.
- 서비스는 최종 보고서까지 자동 생성하는 `CLI 기반` 서비스로 정의한다.
- 비교 대상은 `LG에너지솔루션`과 `CATL`로 고정한다.
- 사용자 입력은 최소화한다.
- 최종 출력은 `한국어 Markdown + PDF`로 생성한다.
- 수집 방식은 `RAG + 웹 검색` 혼합형으로 설계한다.
- 기본 분석 근거는 로컬 코퍼스로 하고, 웹 검색은 보완 용도로만 사용한다.
- 멀티 에이전트 구조는 `Distributed pattern`을 적용한다.
- LG용 에이전트 라인과 CATL용 에이전트 라인을 각각 분리하고, 마지막에 통합 평가한다.
- 실패 시 제한된 재시도 후 부분 보고서를 생성한다.
- 중간 산출물은 모두 저장한다.
- LLM 제공자는 `OpenAI`로 고정한다.
- 기본 모델은 `gpt-4o-mini`를 우선 사용한다.
- 임베딩 모델은 오픈소스를 사용한다.
